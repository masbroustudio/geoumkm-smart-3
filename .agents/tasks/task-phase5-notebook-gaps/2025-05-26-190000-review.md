# Notebook-derived analytics surfaces: DBSCAN, model performance, counterfactuals, and model documentation

This change fills the gap between the ML notebooks and the dashboard by exposing four categories of notebook output that were previously invisible to end users: DBSCAN spatial clustering results, model performance metrics with visual gauges, per-UMKM counterfactual improvement recommendations, and a dedicated model documentation page. The approach is entirely static — hardcoded data files in `lib/` feed the UI components, matching the existing pattern used across the dashboard. The AI chat panel gains four new topic handlers to answer questions about these surfaces.

Watch for:
- **AUC-ROC mislabeled on gauge** (confirmed): the first gauge is labeled "Location Scoring Model" but displays AUC-ROC 0.8303, which belongs to the Credit Risk model per the data file and model-docs page.
- **SVG gauge math assumes 0-1 range** (confirmed): the `strokeDasharray` calculation multiplies the metric value directly by the circle circumference (314.16). For the R2 gauge showing 0.9999, this renders correctly, but the approach breaks if any metric exceeds 1.0 or needs a different visual range.
- **Counterfactual bar widths treat raw scores as percentages** (confirmed): the progress bar uses `style={{ width: \`${cf.current}%\` }}` directly, meaning any `target` value above 100 would overflow.

## High-level view

The clustering page gains a K-Means/DBSCAN tab toggle. The refactoring extracts the existing K-Means view into a separate component and adds a new DBSCAN component displaying summary stats, a top-10 densest clusters table, and a kabupaten density bar chart. The DBSCAN noise ratio is displayed as "0.31%" — this is suspiciously low for DBSCAN and may represent a unit confusion between 0.31 (ratio) and 0.31% (percentage).

The credit scoring page appends a Model Performance section with three SVG circular gauges (AUC-ROC, KS Statistic, R2) and a model comparison table. The gauge labeling attributes AUC-ROC to "Location Scoring Model" when it actually belongs to the Credit Risk model, and the comparison table duplicates the same AUC-ROC value across both models.

The UMKM detail page adds a "How to Improve" card showing counterfactual analysis — per-UMKM before/after progress bars indicating which features to change and the predicted impact. The progress bars use raw data values directly as CSS percentage widths with no clamping.

The model documentation page at `/model-docs` is a new standalone page with model cards, business impact, pipeline overview, limitations, and a deployment checklist.

The AI chat gains handlers for `model performance`, `dbscan`, `counterfactual`/`bagaimana meningkatkan`, and `model card`/`dokumentasi model` queries. The keyword-matching ordering means queries containing "score" will hit the existing credit handler before reaching the model performance handler — a fragile but intentional priority system.

## AUC-ROC gauge attribution error on Credit Scoring page

The Model Performance section renders three gauges. The first uses `modelMetrics.auc_roc` (0.8303) and is labeled "AUC-ROC" with sub-label "Location Scoring Model":

```tsx
<p className="mt-3 text-sm font-medium text-slate-300">AUC-ROC</p>
<p className="text-xs text-slate-500">Location Scoring Model</p>
```

Both the `model-performance-data.ts` comparison table and the model-docs page attribute AUC-ROC 0.8303 to the **Credit Risk (Logistic Regression)** model. The Location Scoring model (XGBoost) uses R2 0.9999 and RMSE as its primary metrics. The confusion stems from the `modelComparison` data file assigning `auc_roc: 0.8303` to both the Location Scoring and Credit Risk entries, conflating the two models' metrics. The executive summary — where AUC-ROC 0.8303 is specifically the credit risk model's discrimination metric — is the authoritative source.

## DBSCAN noise ratio unit concern

The data file stores `noise_ratio: 0.31` and the component renders it as `{dbscanSummary.noise_ratio}%`, producing "0.31%". DBSCAN typically produces noise ratios between 5-30% depending on epsilon and data density. A noise ratio of 0.31% (31 out of 10,000 points classified as noise) would imply an unusually generous epsilon that captures nearly everything into clusters. If the notebook actually outputs 0.31 as a fraction (meaning 31%), the display should either multiply by 100 or remove the `%` suffix and show the raw ratio.

## Chat handler keyword overlap

The new handlers in `getEnhancedResponse` are positioned after the existing credit/score and cluster handlers in the if-else chain. A user asking "bagaimana performa model scoring?" would match `'score'` in the credit handler (which runs first) rather than reaching the model performance handler. This is the same priority-based pattern used throughout — the comment documents it — but it means certain phrasings of model performance questions will get credit-focused responses instead.
