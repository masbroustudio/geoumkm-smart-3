# Next.js 14 frontend scaffold and repository reorganization for GeoUMKM v4.0

This change replaces the monolithic `index.html` landing page with a full Next.js 14 App Router project, restructures the repository so ML content lives under `ml/`, adds comprehensive Azure deployment documentation, and rewrites the GitHub Actions workflow for the new frontend. The approach is "scaffold first": all dashboard pages are placeholders, auth forms have no backend wiring, and the chat panel renders static demo messages.

Watch for: **Untracked `lib/utils.ts`** breaks the build in CI because it was never `git add`-ed (confirmed). **`output: 'standalone'` in `next.config.js` is incompatible** with Azure Static Web Apps which expects static export or the default `.next` build artifacts, not a standalone Node.js server (confirmed). **Double Google Fonts import** loads Inter via both a CSS `@import` in `globals.css` and `next/font/google` in the root layout, shipping unnecessary network requests (confirmed).

## High-level view

The frontend skeleton uses Next.js route groups to separate concerns: `(landing)` for the public marketing page, `(auth)` for login/register, and `(dashboard)` for the authenticated app shell. All seven dashboard pages are identical "Coming Soon" placeholders. The chat panel and auth forms are UI-only—they render but do nothing.

The CI pipeline has a structural mismatch: the workflow deploys to Azure Static Web Apps, but `next.config.js` uses `output: 'standalone'` which produces a Node.js server binary, not static assets. Azure Static Web Apps cannot run a standalone server. The workflow will deploy an empty or broken site.

The `<html>` tag is hardcoded to `className="dark"`, making the light-mode CSS variables and their corresponding `.glass`/`.glass-card` variants unreachable. All component styling uses hardcoded dark-palette Tailwind classes (`text-slate-400`, `bg-slate-800/80`, `border-slate-700`) rather than theme-aware variables, so enabling light mode later will require touching nearly every component.

## Next.js output mode vs. Azure Static Web Apps deployment

The `next.config.js` sets `output: 'standalone'`, which tells Next.js to produce a self-contained Node.js server in `.next/standalone/`. The GitHub Actions workflow then deploys with `output_location: ".next"` to Azure Static Web Apps. Static Web Apps is a CDN-backed static hosting service—it cannot execute a Node.js server process. The workflow will either upload the wrong artifacts or fail at runtime.

Two paths forward: if the app is truly static (no server-side data fetching, no API routes in the Next.js app itself), switch to `output: 'export'` and set `output_location` in the workflow to `out/`. If SSR is needed later, deploy to Azure App Service or Azure Container Apps instead of Static Web Apps, or use Static Web Apps' built-in Next.js hybrid rendering (which requires removing `output: 'standalone'` and letting their build integration handle it with `skip_app_build: false`).

## Untracked `frontend/lib/utils.ts`

The Sidebar component has `import { cn } from "@/lib/utils"` on line 18. The file exists on disk—a standard `clsx` + `tailwind-merge` utility:

```typescript
import { clsx, type ClassValue } from "clsx";
import { twMerge } from "tailwind-merge";

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

`git ls-files --error-unmatch frontend/lib/utils.ts` confirms the file is untracked. A fresh clone (as CI does) won't have this file, and the build will fail with a module resolution error.

## Dual font loading

The root layout uses `next/font/google` to self-host Inter:

```typescript
const inter = Inter({ subsets: ["latin"] });
<body className={inter.className}>
```

Simultaneously, `globals.css` line 1 fires a render-blocking request to Google's CDN:

```css
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@300;400;500;600;700;800;900&display=swap');
```

The two systems will fight over which Inter is applied. The `next/font` approach is superior (self-hosted, subset, no FOUT); the CSS `@import` should be removed.

## Auth forms as dead-end UI

Both `/login` and `/register` pages render forms with `onSubmit={(e) => e.preventDefault()}`. The inputs are uncontrolled (no `value`/`onChange` state), so there's no way to extract user input even if a handler were wired up later. The register page shows "Minimal 8 karakter" as placeholder text but enforces nothing. Unlike the dashboard pages which clearly display "Coming Soon" badges, the auth pages present themselves as functional forms that silently discard all input.

## Hardcoded dark mode paint

The `<html>` element uses a static `className="dark"` with no toggle mechanism or `prefers-color-scheme` media query. The CSS defines `:root` (light) variables alongside `.dark` overrides, but only dark is active. More critically, the component JSX doesn't use the CSS variables for text/background colors—it uses hardcoded Tailwind classes like `text-slate-400`, `bg-slate-800/80`, `border-slate-700` throughout every component. These are dark-palette literals, not theme-adaptive. Switching to light mode later isn't a matter of toggling a class; it requires rewriting color classes in every file.
