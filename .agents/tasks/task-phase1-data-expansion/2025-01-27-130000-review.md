# Full kecamatan data expansion to 595 locations

The frontend static data grows from 20 sample entries to all 595 kecamatan in West Java, sourced from `ml/data/knowledge_base/location_profiles.json`. The map, comparison radar chart, and kabupaten filter now cover the full dataset. Cluster names in `knowledge-base.ts` are corrected to meaningful labels ("Suburban Growth Hub", "Transitional Developing") instead of the confusing suffixed duplicates from the ML pipeline. A manual setup checklist document is added.

**Watch for:** Duplicate kecamatan names across kabupaten cause React key collisions and incorrect comparison lookups (confirmed). The kabupaten filter expands to 27 options but the `recommendData` it filters only covers 3 kabupaten, so most filter selections return empty results (confirmed). Bundle size grows ~130KB of inline data with no lazy-loading path (confirmed).

## High-level view

Both `kecamatanMapData` (coordinates + scores) and `kecamatanDetailData` (multi-dimensional metrics) are generated from the ML knowledge base and embedded at build time as TypeScript array literals, consistent with the project's static-export architecture.

The comparison dropdowns now render 595 options each, but use `key={k.kecamatan}` as the React key. Since 35 kecamatan names appear in multiple kabupaten, this produces duplicate keys and means `find`-by-name lookups return the wrong entry for the second+ occurrence. The kabupaten filter expansion from 6 to 27 options is cosmetically correct but functionally disconnected from the recommendation dataset.

The cluster name fix resolves a real confusion in the ML output where clusters 2 and 4 were named "Urban Digital Leaders (2)" and "High-Risk Underserved (4)" — just the cluster 0/3 names with a numeric suffix. The new names diverge from what the ML source files still label these clusters, creating a drift risk for future pipeline runs.

## Duplicate keys and lookup ambiguity in the comparison feature

The comparison radar chart on the Location Intelligence page lets users pick two kecamatan from `<select>` dropdowns populated by `kecamatanDetailData.map((k) => <option key={k.kecamatan} ...>)`. With 595 entries, 35 kecamatan names repeat across different kabupaten (Cibeureum appears in Kab. Kuningan, Kota Sukabumi, and Kota Tasikmalaya, for example).

The lookup logic `kecamatanDetailData.find((k) => k.kecamatan === compareA)` always returns the first array match. If a user selects "Cibeureum (Kota Tasikmalaya)" from the dropdown, the `value` stored is just `"Cibeureum"`, and the `find` call returns the Kab. Kuningan entry instead. The radar chart then shows incorrect data with no indication of the mismatch.

The kecamatan detail page at `/kecamatan/[name]` uses kabupaten as a secondary discriminator in its lookup, but the comparison dropdowns do not.

## Kabupaten filter coverage gap

The location-intelligence page expands `kabupatenOptions` from 6 to 27 entries. The filter applies to `recommendData`:

```typescript
const filtered = recommendData.filter((item) => {
  if (kabupaten !== 'Semua' && item.kabupaten !== kabupaten) return false;
  return true;
});
```

But `recommendData` only contains entries for Kota Bekasi, Kota Depok, and Kota Bandung (11 entries total). Selecting any of the 24 other kabupaten produces an empty array. The page renders the table with zero rows — no empty state, no message. Users see an expanded filter and assume data exists for their region.

## Bundle size and static-export trade-off

`static-data.ts` grew from ~35KB to ~168KB. The file is imported by at least three pages (overview, location-intelligence, kecamatan detail), and the map/detail arrays are only needed by two of those. A module split would let Next.js code-split more effectively.

The 595-entry map array renders 595 Leaflet markers simultaneously on the overview page. Whether this causes rendering performance issues depends on the map library's clustering configuration, which wasn't changed in this diff.

## Cluster naming reconciliation

The knowledge-base source (`cluster_summaries.json`) names cluster 2 as "Urban Digital Leaders (2)" and cluster 4 as "High-Risk Underserved (4)" — placeholder names from the ML pipeline where the labeling step only assigned meaningful names to 3 of 5 clusters. The frontend renames these in `knowledge-base.ts`, but the ML source files aren't updated in sync. Any automated pipeline that generates API responses or reprocesses the knowledge base will emit the old names, creating a frontend/backend mismatch.
