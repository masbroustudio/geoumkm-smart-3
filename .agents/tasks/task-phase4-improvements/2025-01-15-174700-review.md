# i18n, PWA support, and auth scaffolding for GeoUMKM dashboard

Phase 4 adds three capabilities: a Bahasa/English toggle via a LanguageProvider context, Progressive Web App scaffolding (manifest, service worker, offline page), and a mock authentication provider with an AuthGuard component not yet wired into the layout. The i18n system follows the existing ThemeProvider pattern and covers landing page, sidebar, dashboard overview, and chat panel. The analytics wrapper and service worker are production-ready in structure but use placeholders (console.log) until real secrets are configured.

**Watch for:** The service worker caches navigation responses with a cache-first strategy, meaning stale HTML can be served indefinitely after a deploy without a cache-busting version bump (confirmed). The `isEnabled` flag in analytics.ts is evaluated at module load time in a way that may always be false in a static export since `process.env` substitution depends on the build-time presence of the variable (likely). The i18n test for `t()` returning correct translations asserts the same value for both locales on the `sidebar.overview` key because both id.ts and en.ts use identical English strings for sidebar labels (confirmed) — the test doesn't actually prove locale-switching changes the output for that key.

## High-level view

The i18n system is a lightweight context provider exposing a `t` object (the full translation dictionary for the active locale) rather than a traditional `t('key')` lookup function. Components get compile-time type safety on translation keys, but the entire translation tree is always present in the client bundle. For ~200 keys this is negligible; the pattern won't scale to thousands of keys without code-splitting.

The service worker takes a cache-first approach for all same-origin requests (not just static assets). Combined with `skipWaiting` and `clients.claim`, newly deployed HTML pages will be served from the stale cache until the browser re-fetches the service worker (up to 24 hours). There is no cache versioning strategy beyond the hardcoded `geoumkm-v1` string.

The auth scaffolding is intentionally inert — `AuthProvider` is not in the component tree for any route, and `AuthGuard` must be manually wired in. The mock state has no persistence, so "logging in" lasts only until page reload.

The analytics wrapper evaluates `process.env.NEXT_PUBLIC_APPINSIGHTS_KEY` at module load time. In a Next.js static export, `NEXT_PUBLIC_*` variables are inlined at build time. If the variable is absent during `next build`, the compiled output resolves to `!!""` (always false), and setting it at runtime in the hosting environment won't help.

## Service worker caching strategy and deploy lifecycle

The service worker in `public/sw.js` caches the app shell URLs on install and then uses cache-first for all same-origin fetches:

```js
caches.match(request).then((cached) => {
  if (cached) return cached;
  return fetch(request).then((response) => {
    // ...cache the response
  });
});
```

Navigation requests (HTML pages) hit this same path. After a deploy, the old cached HTML is served immediately. The `CACHE_NAME` is hardcoded to `'geoumkm-v1'` — there's no automated mechanism to bump it on deploy. The `activate` handler cleans up old cache names, but since the name never changes, old entries persist indefinitely.

For a static export deployed to Azure Static Web Apps, users could see a previous version's HTML until the browser re-fetches `sw.js` (up to 24 hours per HTTP caching rules, unless the hosting sets short `Cache-Control` headers on `sw.js` itself). Network-first for navigation with cache fallback would be more appropriate for an app that deploys frequently.

## Analytics build-time evaluation in static exports

```ts
const isEnabled = typeof process !== 'undefined' && !!process.env.NEXT_PUBLIC_APPINSIGHTS_KEY;
```

In Next.js static export (`output: 'export'`), all `NEXT_PUBLIC_*` variables are string-replaced at build time. If the build environment doesn't have `NEXT_PUBLIC_APPINSIGHTS_KEY` set, the compiled JS will contain `!!""` which is `false`. The documentation in `MANUAL_SETUP_NOTES.md` section K.3 says to add the key to "Azure Static Web Apps environment variables" for production — but those runtime env vars are not accessible to pre-built static client-side JavaScript. The key must be present during `pnpm build`.

## i18n sidebar labels: same strings in both locales

The Indonesian translation file (`id.ts`) uses English strings for the sidebar:

```ts
sidebar: {
  overview: 'Overview',
  creditScoring: 'Credit Scoring',
  // ...all English
}
```

And for most dashboard keys (`Total UMKM`, `Avg Score`, `High Risk Areas`, etc.). Only the theme mode labels differ: `Mode Terang` / `Mode Gelap`. Landing page text does differ between locales. Toggling language in the dashboard sidebar produces no visible change for most labels.

This also means the i18n test that checks `t.sidebar.overview` across locales is not verifying that locale switching produces different output — both locales return "Overview". The separate test using `landing.hero.title` does properly validate this.

## `<html lang>` doesn't track the active locale

The root layout hardcodes:

```tsx
<html lang="id" className="dark" suppressHydrationWarning>
```

The `LanguageProvider` tracks locale in React state but never updates `document.documentElement.lang`. When the user toggles to English, the document still declares itself as Indonesian to screen readers. The ThemeProvider already demonstrates the pattern of updating `document.documentElement` attributes from a `useEffect` — the same approach could set the `lang` attribute.
