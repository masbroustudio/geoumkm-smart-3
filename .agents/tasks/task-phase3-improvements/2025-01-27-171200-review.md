# GeoUMKM Phase 3: Dynamic Simulator, UMKM Directory, Azure OpenAI, Heatmap Layer

Four features land in one batch: a custom what-if API with linear-approximation scoring, a 50-profile UMKM directory page, an Azure OpenAI chat integration with keyword-matching fallback, and a toggle-able heatmap on the overview map. The implementation follows the existing static-export constraint throughout, uses query params for the UMKM detail route, and degrades gracefully when the API or Azure credentials are absent.

Watch for:
- **Negative score not capped** (confirmed) — the what-if formula allows scores to go below 0 when deltas are negative, only caps at 100 on the high end.
- **Typing effect race on rapid send** (confirmed) — a brief window exists between `setIsLoading(false)` and the typing effect starting where a second message could overwrite the first response before it's appended to history.
- **Conversation history ignored by Azure OpenAI** (confirmed) — the frontend sends `history` but the API handler never passes it to the OpenAI messages array. Multi-turn context is lost.

## High-level view

The what-if custom simulator duplicates the linear formula in two places (API and client-side fallback) with identical coefficients and cap logic. The formula omits a floor at 0, meaning negative deltas can produce scores below zero in both paths.

Azure OpenAI integration is surgically scoped: one `callAzureOpenAI` function wraps a single fetch with a 15-second abort timeout, and the entire path returns `null` on any failure, dropping into the pre-existing keyword matcher. The frontend typing effect simulates streaming by revealing words on an interval, but the transition from "loading" to "typing" state has a one-render gap exploitable by fast users.

The UMKM directory bundles 50 profiles as static data (~15-20KB gzipped). The `MapInner` component now imports this data to provide popup links, coupling the map chunk to the full UMKM dataset.

The heatmap is a proportional-symbol map (not kernel density) using `CircleMarker` with score-derived radius and color, toggled via state on the overview page.

<summary>Issues (6)</summary>

1. **Score floor missing in what-if formula** — Both API and client-side fallback cap scores at 100 but never floor at 0. Negative deltas (e.g., infrastructure -30, internet -30) can produce negative scores displayed in the UI. Add `Math.max(0, ...)` alongside the existing `Math.min(100, ...)`.
2. **Typing effect race condition** — `isLoading` is set to `false` before `setTypingFullText` triggers the effect that sets `isTyping = true`. In that one-render gap, a suggestion click or fast Enter could fire `handleSend`, overwriting `typingFullText` and losing the first response. Move `setTypingFullText` before `setIsLoading(false)`, or set `isTyping` synchronously.
3. **Conversation history not passed to Azure OpenAI** — The API accepts `history` in the request body but constructs the OpenAI messages array with only `[system, user]`. The model has no multi-turn context despite the frontend displaying a conversation thread. Pass at least the last N history messages to the model.
4. **Client/API formula drift risk** — The linear formula `score + (infra * 0.66) + (internet * 0.33) + (bank * 0.01)` is hardcoded in two places with no shared constant or comment referencing the ML notebook that derived these coefficients. Future coefficient changes must be synchronized manually.
5. **No input validation on delta ranges in API** — The `whatif-custom` endpoint validates that `target_kabupaten` exists but does not validate that deltas are within expected ranges. A client could send `infrastructure_delta: 9999` and get uncapped results.
6. **Context truncation at character boundary** — `buildKnowledgeContext` slices the context string at 1000 characters, which can cut mid-sentence or mid-example, potentially confusing the model. Truncate at a sentence or example boundary instead.

## What-if custom simulator: duplicated formula without floor

The core formula in `api/src/functions/whatif-custom.ts` and `frontend/lib/api.ts` (client fallback):

```typescript
const rawNew = scoreBefore +
  (infrastructure_delta * 0.66) +
  (internet_delta * 0.33) +
  (bank_distance_delta * 0.01);
const scoreAfter = Math.min(100, rawNew);
```

When all three deltas are at their minimum (-30, -30, -50), the formula subtracts `(30 * 0.66) + (30 * 0.33) + (50 * 0.01) = 30.2` points. A UMKM with a starting score of 15 would end up at -15.2, displayed as a negative value in the UI. The fix: `Math.max(0, Math.min(100, rawNew))`.

The API also does no range validation on incoming deltas — it trusts the frontend slider bounds. A direct API call with extreme values would exacerbate the negative-score problem.

## Chat typing effect and message loss on rapid send

The typing flow: response arrives → `setIsLoading(false)` → `setTypingFullText(responseText)` → useEffect fires on next render → `setIsTyping(true)` → interval reveals words → completion appends message.

The gap between `setIsLoading(false)` and `setIsTyping(true)` is one React render cycle. During this gap, `handleSend`'s guard `if (!userMessage || isLoading || isTyping) return` passes because both `isLoading` and `isTyping` are false. A suggestion click in that window would set a new `typingFullText`, the cleanup function would clear the first interval, and the first response would never be appended to `messages`.

## Azure OpenAI: single-turn, blunt truncation

The `callAzureOpenAI` function sends:

```typescript
messages: [
  { role: "system", content: systemPrompt },
  { role: "user", content: message },
]
```

The `ChatMessage` type includes `history?: { role: string; content: string }[]` and the frontend sends it, but it's never used in the OpenAI call construction. Every message is treated as a fresh conversation.

`buildKnowledgeContext` constructs a context string from cluster profiles and persona examples, then truncates: `context.slice(0, 1000)`. With 5 clusters and 2 example Q&A pairs, the string typically exceeds 1000 chars, so truncation happens inside the second example's response text. The model receives a partial example with no closing.

## MapInner coupling to UMKM data module

`MapInner.tsx` now imports `umkmDetailData` at the top level. This means the Webpack/Next.js bundler includes the entire `umkm-detail-data.ts` module in the map chunk. For each of 595 map markers, it runs `umkmDetailData.find(u => u.kecamatan === item.kecamatan)` — O(595 * 50) per render. Since only 50 of the 595 kecamatan have a matching UMKM, most iterations find nothing. A pre-built `Map<string, UmkmDetail>` keyed by kecamatan would eliminate the repeated linear scan and the module already exports `umkmDetailMap` (keyed by ID, not kecamatan) — a second map keyed by kecamatan would be the clean fix.
