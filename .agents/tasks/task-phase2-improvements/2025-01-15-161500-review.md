# Phase 2: Dark mode, testing, error boundaries, and API workflow (v2)

Four features shipped together: CSS-variables-based dark/light theming with a sidebar toggle, Vitest test suite (25 tests across 5 files), Skeleton and ErrorBoundary components wired into the dashboard layout, and a GitHub Actions workflow for Azure Functions deployment. The two blocking issues from the v1 review — ErrorBoundary infinite crash loops and missing PR validation on the API workflow — have both been resolved.

**Watch for:** Skeleton component uses Tailwind `dark:` variant while all other modified components use CSS variables, creating a style-pattern divergence within the same feature set (confirmed). KPICard tests are coupled to exact Tailwind class names rather than behavior (confirmed). The API workflow `package: "./api"` still ships the full source tree — acceptable if the Azure Functions runtime handles it, but worth verifying on first deploy (possible).

**Verdict**: APPROVED

## High-level view

The ErrorBoundary now increments a `resetKey` counter on retry, wrapping children in `<React.Fragment key={resetKey}>`. This forces React to unmount and remount the subtree, breaking the infinite crash loop flagged in v1.

The API workflow now triggers on both `push` and `pull_request` events, with the deploy step gated by `if: github.event_name == 'push'`. PRs touching `api/**` get install + build validation without deploying.

The remaining concerns are non-blocking: a style inconsistency in the Skeleton component, test assertions coupled to class names rather than rendered behavior, and the previously-noted `package: "./api"` path which is an operational concern for first deployment rather than a code defect.

<details>
<summary>Issues (3)</summary>

1. **Skeleton uses Tailwind dark variant instead of CSS variables** — `dark:bg-slate-700/50` diverges from the CSS variable pattern used by Sidebar, KPICard, and ErrorBoundary in the same PR. Future contributors won't know which pattern to follow. Consider using `bg-[var(--border-color)]` or similar for consistency.
2. **KPICard trend tests coupled to class names** — Tests assert on `.text-emerald-400` and `.text-red-400` via `container.querySelector`. A Tailwind color rename or component refactor breaks the tests without breaking functionality. Prefer testing for presence of the SVG icon element or an aria-label.
3. **API workflow package path** — `package: "./api"` deploys the full source directory including `src/`. If the Azure Functions runtime only needs compiled output, this ships unnecessary files. Verify on first deployment; no code change needed unless it fails.

## Skeleton style divergence

The Skeleton component uses `bg-slate-200 dark:bg-slate-700/50` — Tailwind's class-based dark mode. Every other component modified in this PR (Sidebar, KPICard, ErrorBoundary, dashboard layout) uses CSS custom properties like `text-[var(--foreground)]` or `bg-[var(--sidebar-bg)]`. Both mechanisms work because `.dark` is toggled on the HTML element, but the inconsistency within a single feature set will confuse contributors about which pattern to adopt for new components.

## Test coupling to implementation details

The KPICard trend tests select elements by Tailwind class (`.text-emerald-400`, `.text-red-400`). If the trend colors are refactored to CSS variables (consistent with the theming approach in this PR), or if Tailwind's color palette is customized, these tests break without any user-visible regression. A more resilient approach would test for the presence of the arrow SVG icon itself or add `data-testid` attributes to the trend indicators.
