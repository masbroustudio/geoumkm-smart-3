# Phase 2: Dark mode, testing framework, error boundaries, and API deployment workflow

Four features shipped in one batch: CSS-variables-based theming with a sidebar toggle, a Vitest test suite (25 tests across 5 files), reusable Skeleton/ErrorBoundary components wired into the dashboard layout, and a GitHub Actions workflow for Azure Functions API deployment. The approach deliberately keeps dark mode changes surgical by using CSS custom properties at `:root`/`.dark` scope rather than rewriting every component's Tailwind classes.

**Watch for:** ErrorBoundary retry resets state but doesn't remount children, so the same crash-causing data will re-trigger the error immediately (confirmed). Theme context tests pass but rely on jsdom's `localStorage` stub which doesn't prove the flash-prevention script works in a real browser (likely). The API workflow deploys the entire `./api` directory including source files — it may need a dist-only package step depending on Azure Functions expectations (possible).

## High-level view

The theming system defines CSS variables in `:root` (light) and `.dark` (dark) scopes. The sidebar reads `useTheme()` and renders a toggle. Because the project already had a `ThemeProvider` and inline flash-prevention script, this change only needed the variable palette and swapping hardcoded colors in Sidebar and KPICard. Other components still use Tailwind `dark:` variants directly, which work because `.dark` is toggled on `<html>`.

The test suite uses Vitest with jsdom and `@testing-library/react`, exercising `cn()` conflict resolution, CSV generation with quote escaping, API envelope unwrapping and fallback behavior, theme toggle persistence, and KPICard trend rendering.

The ErrorBoundary wraps `{children}` inside `<main>` in the dashboard layout, so a page-level crash shows a fallback without taking down the sidebar or chat panel. The retry mechanism has a flaw: it re-renders the same children without changing any inputs.

The API deployment workflow mirrors the frontend workflow's structure but uses `Azure/functions-action@v1` with a publish profile secret. It only triggers on pushes to `main` — no PR validation gate.

## ErrorBoundary retry semantics

The `handleReset` method sets `{ hasError: false, error: null }`, which causes `render()` to return `this.props.children` again. If the error was caused by the children's own render logic (a null-access on data, a missing prop), nothing has changed — the same component tree re-renders and immediately throws again. The user sees a flicker.

```tsx
handleReset = () => {
  this.setState({ hasError: false, error: null });
};
```

A common fix is to assign a `key` to a wrapper inside the boundary that increments on reset, forcing React to unmount and remount. Alternatively, the retry button could trigger a navigation or data refetch that changes the conditions that caused the crash.

This matters most because the boundary wraps `{children}` in the dashboard layout — every page route renders inside it. A crash in any page component creates an infinite crash loop until the user navigates away via the sidebar.

## API deployment: package path and missing PR gate

The `deploy-api.yml` sets `package: "./api"` which tells `Azure/functions-action` to deploy the contents of that directory. After `pnpm build` runs, the action uploads the entire `./api` tree. Whether this works depends on the Azure Functions project structure — if `host.json` and compiled output live at `./api` root level, it's correct. If the build emits to `./api/dist` and the function bindings reference that path, the action should point to `./api/dist` instead.

The frontend workflow includes a `pull_request` trigger so PR changes are validated before merge. The API workflow lacks this — a broken API build won't fail CI until after the PR is merged. Adding a `pull_request` trigger with just the build steps (no deploy) would catch build failures earlier.

## Skeleton style inconsistency

The Skeleton component uses `dark:bg-slate-700/50` (a Tailwind dark variant) rather than a CSS variable. The same PR establishes a CSS variable pattern for Sidebar and KPICard. Not a bug — `.dark` is toggled on the HTML element so both approaches work — but it's a style divergence within a single feature set that could confuse future contributors about which pattern to follow.

## Test gaps

The KPICard trend-direction tests assert on CSS class presence (`.text-emerald-400`, `.text-red-400`) via `container.querySelector`. These are coupled to the current class names — a color rename in Tailwind config or the component would break the tests without breaking functionality.

The theme context tests don't cover the `mounted` guard (`if (!mounted) return <>{children}</>` path). Before the useEffect fires, children render outside the context provider, receiving the default context value with a no-op `toggle`. No test proves children still render correctly during that pre-mount phase.

The CSV BOM prefix (`\uFEFF` for Excel compatibility) isn't explicitly asserted in the quote-escaping test, even though it reads the blob content. A future refactor could remove the BOM without failing any test.
